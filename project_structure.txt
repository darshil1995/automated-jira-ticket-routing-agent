agents/ — each agent (Triage, Resolution, QA) gets its own file. Keeps concerns separated and makes it easy to swap one agent without touching the others.
utils/ — shared helpers like the mock JIRA writer and the knowledge base builder live here.
tickets/ — mock JIRA output JSON files get saved here during local testing.
sample_data/ — a few realistic alert strings you can use to test the pipeline.
lambda_handler.py — the AWS Lambda entry point. Stays thin, just receives the event and calls the graph.
graph.py — the LangGraph wiring that connects all three agents.


Virtual environment — isolates this project's dependencies from everything else on your machine. Without it, installing faiss-cpu here could break another project that needs a different version.
requirements.txt — when you deploy to Lambda or a colleague clones the repo, they run pip install -r requirements.txt and get the exact same versions you tested with. No surprises.
.env + python-dotenv — keeps secrets out of your code. Locally it reads from the file; in Lambda you set the same variable names in the AWS console under "Environment variables" and load_dotenv() simply does nothing.
config.py — one source of truth for all configuration. Every agent imports from config import LLM_MODEL rather than calling os.getenv() scattered across files. The _require() helper means the app fails loudly at startup with a clear message if a variable is missing, rather than silently failing mid-pipeline when an LLM call is attempted.
MOCK_MODE — lets you run and test the entire pipeline for free during development. When it's True, agents return hardcoded realistic responses instead of calling OpenAI. You only flip it to False when you're ready to test with real LLM calls.


@tool decorator — tells LangChain this function is callable by an LLM agent as a tool. When we connect this to LangGraph in a later step, the agent can decide to call it mid-reasoning rather than needing you to call it manually.
MOCK_MODE branch — the very first thing run_triage checks. This means you can run the entire pipeline a hundred times while building it without touching your OpenAI balance.
temperature=0 — classification tasks need consistency. The same issue should always get the same category. Temperature 0 removes randomness from the LLM output.
Structured JSON prompt — instead of asking the LLM to "describe the issue", we tell it exactly what keys to return. This makes the output machine-readable and safe to pass into the next agent via LangGraph state.
Module-level logger with __name__ — every log line from this file will say agents.triage_agent as its source. When you have three agents all writing to CloudWatch, this makes it immediately clear which agent produced which log line.

What the QA Agent adds
Two-layer review — regex catches obvious violations (email addresses, AWS keys, internal IPs) for free before any LLM call. The LLM then handles nuanced cases regex can't — indirect PII, vendor name references, speculative language. In mock mode only the regex layer runs, keeping local testing completely free.
Static override — if regex finds a violation but the LLM somehow approves the ticket anyway, the agent overrides the LLM's decision. Deterministic checks always win over probabilistic ones for compliance.
Clean final output — the final_ticket field inside the QA result is what gets written to JIRA. If violations were found, it contains the sanitised version with [REDACTED-PII] placeholders. If clean, it's the original resolution unchanged.


What this handler adds
Two trigger types in one function. The _detect_event_type() function inspects the event shape and routes it correctly — an HTTP POST from API Gateway and an S3 file upload both end up calling run_pipeline() with an issue string. The pipeline has no idea which trigger fired.
Three-tier error handling. ValueError means the caller sent bad input — return 400. RuntimeError means infrastructure failed — return 500. Anything else is unexpected — log the full traceback to CloudWatch and return 500. This distinction matters in production because it tells you whether to fix the caller or fix your code.
Request ID tracing. Every log line includes the Lambda aws_request_id. In CloudWatch you can filter by this ID and see every log line for a single invocation — across all three agents — in one view.
